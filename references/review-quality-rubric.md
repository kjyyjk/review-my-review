# 코드 리뷰 품질 루브릭

## 코멘트 유형 분류

| 유형 | 설명 | 예시 |
|------|------|------|
| 질문형 | 의도/이해 확인을 위한 질문 | "이렇게 구현하신 이유가 있나요?" |
| 제안형 | 더 나은 방법을 제안 | "X 대신 Y를 쓰면 Z가 좋아져요" |
| 칭찬형 | 잘한 부분을 명시적으로 칭찬 | "이 패턴 덕분에 가독성이 높아졌네요 👍" |
| 지적형 | 문제 지적 (수정 방향 없음) | "이 부분이 이상해요" |

---

## 좋은 코멘트 패턴

### 맥락이 있는 질문형
```
❌ "왜 이렇게 했나요?"
✅ "여기서 synchronized를 사용하신 이유가 있나요? 성능 영향이 있을 것 같아서요."
```

### 이유가 있는 제안형
```
❌ "이 방식은 별로예요. 다른 방식을 써주세요."
✅ "userId를 직접 받는 대신 User 객체를 받으면 추후 사용자 정보가 추가될 때 시그니처 변경 없이 확장할 수 있어요."
```

### 구체적인 칭찬형
```
❌ "잘 하셨네요 👍"
✅ "//.\\n.* 패턴으로 커스텀 구분자를 간결하게 처리하셨네요. 직접 파싱하는 것보다 훨씬 읽기 좋아요 👍"
```

### 학습 유도형 (다중 질문)
```
"접근 제어자를 명시하지 않으면 어떻게 동작할까요?
그리고 이 맥락에서 어떤 접근 제어자가 적절할까요?"
```

## 피해야 할 패턴

| 나쁜 예 | 문제점 | 개선 방향 |
|---------|--------|-----------|
| "이 코드 이상해요" | 무엇이 문제인지 불명확 | "X 이유로 Y 문제가 생길 수 있어요" |
| "고쳐주세요" | 무엇을 어떻게 고칠지 없음 | "A를 B로 변경하면 C를 방지할 수 있어요" |
| "왜 이렇게 했나요?" (단독) | 맥락 없는 질문은 방어적 반응 유발 | 왜 묻는지 한 문장 추가 |
| "잘 하셨어요 👍" | 무엇이 왜 좋은지 없음 | 구체적으로 어떤 점이 좋은지 |
| 이유 없는 스타일 강요 | 팀 컨벤션 근거 없음 | "저희 컨벤션에서는 X를 권장해요" |

## 놓치기 쉬운 관점 체크리스트

### 버그/안전성
- [ ] null/empty 처리 누락은 없는가?
- [ ] 0 나누기, 음수 입력 등 경계값 처리는 있는가?
- [ ] race condition 가능성은 없는가?
- [ ] 예외 전파/처리가 적절한가?

### 보안
- [ ] 사용자 입력값 검증이 있는가?
- [ ] 민감 정보가 노출되지 않는가?
- [ ] SQL injection, XSS 등 OWASP 취약점은 없는가?

### 설계
- [ ] 접근 제어자가 적절한가?
- [ ] 레이어 의존 관계가 올바른가?
- [ ] 단일 책임 원칙을 지키고 있는가?

### 테스트
- [ ] 엣지케이스 테스트가 있는가?
- [ ] @DisplayName이 테스트 내용과 일치하는가?
- [ ] happy path만 테스트하고 예외 상황을 빠뜨리지 않았는가?

## 우선순위 태그 (선택적 사용)

필수는 아니지만, 태그를 붙이면 리뷰이가 무엇부터 할지 빠르게 파악 가능:

| 태그 | 의미 |
|------|------|
| `[blocking]` | 머지 전 반드시 수정 |
| `[suggestion]` | 더 나은 방법 제안 (리뷰이 판단) |
| `[nit]` | 취향/스타일 (무시해도 됨) |
| `[question]` | 의도 확인, 토론 |
| `[praise]` | 잘한 부분 칭찬 |
