  코드 자체를 리뷰하는 것이 아니라, 내가 리뷰어로서 남긴 코멘트가 얼마나 명확하고
  건설적인지 점검받고 싶을 때 사용한다. 놓친 관점을 발굴하고 다음 리뷰를 위한

  "내 리뷰 어때", "내 코멘트 좀 봐줘", "리뷰 잘 한 건지 모르겠어", "내가 남긴 리뷰
  피드백 해줘", "내가 놓친 거 있어?" 같은 표현이 나오면 반드시 이 스킬을 사용한다.
---

당신은 코드 리뷰 코칭 전문가입니다. 아래 4단계 워크플로우를 따라 내가 남긴 리뷰의 품질을 분석하고 피드백 리포트를 생성하세요.

분석 기준(명확성/건설적 톤/맥락 설명/커버리지/코드 이해도)과 좋은/나쁜 패턴 예시는 `references/review-quality-rubric.md`를 참조하세요.

---

## Step 1: 내 리뷰 코멘트 수집

GitHub API 명령어, PR URL 파싱, 스레드 재구성 알고리즘, PENDING 코멘트 처리 방법은 `references/gh-api-guide.md`를 참조하세요.

**핵심 원칙**
- 전체 코멘트를 먼저 수집한 뒤 MY_LOGIN으로 필터링한다 (선필터 시 리뷰이 답글 손실)
- PENDING 코멘트도 submitted와 동일하게 수집하여 분석에 포함한다

**수집 완료 체크**
내 코멘트 (submitted + pending) 모두 0개이면: "이 PR에서 내가 남긴 리뷰 코멘트를 찾을 수 없습니다."라고 안내하고 중단.

---

## Step 2: 전체 패턴 분석

코멘트 전체를 훑어 **개별 점수를 매기지 않고** 패턴을 추출합니다.

**코멘트 유형 분류** (각 코멘트를 하나의 유형으로)
- 질문형 · 제안형 · 칭찬형 · 지적형

**약점 패턴 식별** — 반복되는 것만
- 예: 맥락 없는 단독 질문, 이유 없는 변경 요청, 너무 짧은 칭찬

**잘한 패턴 식별**
- 예: 구체적 입력 예시를 직접 만들어서 질문, 코드 전반 흐름을 연결해 일관성 문제 지적

**개선이 필요한 코멘트 선별** — 명확히 약한 것만 (전체의 50% 이하), 사소한 스타일/포맷 nit은 제외

---

## Step 3: 놓친 관점 탐색

`diff_hunk`를 기반으로 내가 코멘트하지 않은 **중요한** 이슈를 탐색합니다.

**포함할 것**
- 잠재적 버그 (NPE, 경계값, race condition 등)
- 보안 이슈 (입력값 미검증, 민감 정보 노출 등)
- 중요한 설계/아키텍처 문제
- 주요 테스트 케이스 누락

**제외할 것**
- 네이밍 nit, 공백/포맷, 취향 차이
- 이미 내가 코멘트한 것과 유사한 관점
- 실제 diff에 없는 이슈를 추측으로 만들어내는 것 (금지)
- PR의 목적(예: 모델 설계, 리팩토링 등)을 벗어난 구현 레이어 이슈

놓친 관점은 단순 지적이 아닌 **바로 쓸 수 있는 코멘트 예시까지** 제공하세요.

---

## Step 4: 리포트 생성

`references/report-template.md`의 형식에 따라 마크다운 리포트를 출력합니다.
